Conversation Analysis Alternative Proposal

对设计文档的重新思考。核心理念：不是"分类然后隐藏"，而是"保留空间关系，调节透明度"。 一个视图解决所有问题，像 X 光片一样——同一张图，调节对比度就能看到不同的层。

一、对设计文档的 Review Critique

设计文档做对了什么

  • "分类而不是删除" — 这个原则完全正确，原始数据不能丢
  • 分析师视角 — 认识到这不只是聊天查看器，而是 AI 行为分析工具
  • provider 差异意识 — 正确区分了 Anthropic 快照 vs OpenAI 增量的差异

设计文档的问题

✗ 问题 1：过度工程化数据模型

  • 新增 TimelineMessageVM 带 6 个新字段 (category, source, visibility, annotations...)
  • 新增 TimelineHiddenGroupVM 联合类型
  • 把展示逻辑的决策（visibility: hidden/primary/secondary）硬编码到数据层
  • Main process 不应该决定"什么该隐藏"——这是 UI 的事

✗ 问题 2：正则分类器是脆弱的

  • classify-message.ts 用正则匹配 <system-reminder> 等模式
  • 每个 AI 客户端的注入格式都不同，且随版本变化
  • 永远在追着上游改——今天是 <system-reminder>，明天可能换格式
  • 误判风险：用户真正输入了 <system-reminder> 会被错误分类

✗ 问题 3：三个 Tab 破坏空间连续性

  • Conversation / Context / Raw 三个 tab 是三个独立视图
  • 切换 tab 后，用户失去"这段注入在对话中的位置"的感知
  • 对于"分析 AI 底层逻辑"的目标，空间位置恰恰是最重要的信息
  • "第 3 条 user message 前被注入了什么"——tab 切换后这个上下文就丢了

✗ 问题 4：忽视已有结构

  • 当前 NormalizedExchange 已经分离了 instructions / inputMessages / tools
  • 问题不在数据模型，在于 timeline assembler 只是 [...input, ...output]
  • 以及 renderer 不区分 block 内容类型
  • 解法应该是增强现有结构，而不是重建一套

二、我的方案：Layered Transparency Core Idea

核心隐喻：X 光模式。同一个视图，同一个空间位置。 不是把信息搬到不同的 tab 里，而是在原地调节"透明度"—— 普通模式看到的是皮肤，X-Ray 模式看到的是骨骼，但它们的空间关系从未改变。

01

一个视图，两种密度

不用 tab 切换。默认模式折叠注入内容为"上下文芯片"；X-Ray 模式在原位展开一切。Inspector 继续作为 Raw 层。

02

Block 级注解，不是 Message 级分类

一条 user message 里可能混着真实输入和 system-reminder。分类粒度应该在 block，不是整条 message。

03

Renderer 决定展示，Main 只提供结构

Main process 负责提取结构信息（这个 block 是文本/是 XML wrapper/是 instructions），Renderer 决定怎么画。

三、默认模式 — Clean Conversation Default View

注入内容不消失，而是折叠成带颜色编码的"上下文芯片"。芯片保留在对话流中原本的位置， 让用户始终感知到"这里有东西被注入了"。点击即可展开查看原文。

五、实现策略 Architecture

5.1 数据模型变更 — 最小化

不重建数据模型。只做两个小扩展：

// 1. 给 NormalizedMessageBlock 加一个可选的 meta 字段 // 注解内容由 timeline assembler 填充 type BlockMeta = { /** 这个 block 是否是客户端框架注入的（非用户原始输入） */ injected?: boolean; /** 注入类型标识（用于 renderer 着色和分组） */ contextType?: 'system-instructions' | 'system-reminder' | 'hook-output' | 'skills-list' | 'meta-prompt'; /** 预估字符数（用于 UI 展示，不需要精确 token 计数） */ charCount?: number; }; // 2. 给 SessionTimeline 加 context summary interface SessionTimeline { messages: NormalizedMessage[]; /** 可选的上下文摘要，由 timeline assembler 填充 */ contextSummary?: { instructionChars: number; reminderChars: number; toolCount: number; injectedBlockCount: number; }; }

就这些。不新增 VM 类型，不新增 visibility/category/source 字段。

5.2 改造路径

Phase 1: Block-level annotation (Main process) ├── derive-title.ts 已有的 NOISE_PATTERNS → 复用 ├── timeline-assembler.ts (Anthropic) │ └── 遍历 inputMessages，对每个 text block： │ ├── 检测 <system-reminder> wrapper → meta.injected = true │ ├── 检测 hook output patterns → meta.contextType = 'hook-output' │ └── 其余保持不变 ├── timeline-assembler.ts (OpenAI) │ └── 同理，但 OpenAI 的注入模式不同（developer message 等） └── contextSummary 在 build() 末尾统计 Phase 2: Smart renderer (Renderer) ├── conversation-view.tsx │ ├── 新增 viewMode state: 'clean' | 'xray' │ ├── 遍历 messages 时，根据 block.meta.injected 决定渲染方式 │ │ ├── clean mode: 渲染为 ContextChip（折叠） │ │ └── xray mode: 渲染为 ContextChip（展开）+ 原文 │ └── 相邻的 injected blocks 自动合并为一个 chip ├── context-chip.tsx (新) │ └── 折叠/展开的上下文指示器 └── context-panel.tsx (新) └── 右侧分析面板：token 组成、注入清单、洞察 Phase 3: Context analysis panel (Renderer) ├── Token composition 柱状图 ├── Injected context 列表（可点击跳转到对话中的位置） └── 自动生成 observations（overhead ratio 等） Phase 4: 回归验证 └── 同设计文档，基于真实历史数据验证

5.3 与设计文档的关键差异

设计文档方案

  • 新增 2 个 VM 类型 + 6 个新字段
  • Main process 做完整语义分类
  • 3 个 Tab（Conversation / Context / Raw）
  • 改动 shared contracts + assemblers + renderer
  • 需要新增 classify-message.ts + build-analysis-timeline.ts

我的方案

  • 给现有 Block 类型加 1 个可选 meta 字段
  • Main process 只做轻量结构检测（已有 pattern 复用）
  • 1 个视图 + Clean/X-Ray 切换 + 可选分析面板
  • Assembler 小改 + Renderer 新增 2 个组件
  • 复用 derive-title.ts 已有的检测逻辑

六、为什么这个方案更好 Rationale

对用户

  • 零学习成本 — 默认视图就是干净的对话，不需要理解 tab 的含义
  • 空间连续性 — 注入内容保留在对话流中的原始位置，切换模式不丢失上下文
  • 渐进发现 — 看到芯片 → 好奇点击 → 理解注入内容 → 打开分析面板 → 全景理解
  • Consumer-grade feel — 像 Arc Browser 的 Boosts，不像 Wireshark 的 filter tabs

对代码

  • 最小改动 — 不重建数据模型，复用已有检测逻辑
  • Renderer owns presentation — 展示决策在 UI 层，不侵入数据层
  • 向后兼容 — meta 字段可选，老数据直接渲染为 clean conversation
  • 可增量交付 — Phase 1 (annotation) 可以单独发，Phase 2 (renderer) 可以单独发

对"分析 AI 底层逻辑"的目标

  • 位置信息是关键 — "这段 CLAUDE.md 是在用户说话之前注入的"，这个空间关系在 tab 切换中会丢失
  • Context Analysis 面板 — 不只是看内容，还能看 token 组成、overhead ratio、注入模式
  • 跨 provider 对比 — 同一个 UI 框架，Anthropic 和 OpenAI 的注入模式差异一目了然
